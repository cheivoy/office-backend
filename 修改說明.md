# 修改說明 — 三個問題修復

本次修改解決你回報的三個問題：寫入返回空白檔案、核對需要持久記錄＋重複/遺漏提醒、檔案預覽沒反應。
以下按問題列出改了哪些檔案、為什麼改、怎麼測。

---

## 問題 1：寫入功能返回空白檔案

### 根本原因
1. 「Nokia 工作天數表 / CHT DK」用的是**伺服器固定範本**（`templates/cht_nokia/`、`templates/cht_dk/`），
   目前 repo 裡是空的 → 找不到範本時程式直接 `continue`，回傳一個沒寫任何資料的檔案。
2. 每個員工寫入失敗都被 `except: pass` 默默吞掉，沒有任何錯誤提示，所以你拿到空白檔也不知道為什麼。

> 寫入邏輯本身是對的（已實測 `write_cht` 能正確填格）。問題在「範本缺失 + 靜默失敗」。

### 改了什麼
- **後端 `services/batch_write_svc.py`**：五個批量寫入函式全部改成回傳 `(bytes, report)`。
  `report` 記錄 `written`（成功幾筆）、`skipped`（跳過的員工 + 原因）、`pm_files_missing`（缺哪些 PM 範本）。
  Nokia 表若**所有範本都不存在**，直接 raise（不再回傳空 zip）。
- **後端 `routers/batch.py`**：把 `report` 用 `X-Write-Report` header（URL-encoded JSON）帶回前端。
- **後端 `main.py`**：CORS `expose_headers` 加上 `X-Write-Report`，讓瀏覽器讀得到。
- **前端 `api.js`**：blob 下載時順便把 `X-Write-Report` 解析並掛在 `blob.writeReport`。
- **前端 `components/WriteModal.js`**：
  - 修掉 `checkedSecs` 是 `Set` 的問題（`JSON.stringify` 會變 `{}`），送出前先轉成陣列。
  - 寫入完成後顯示**寫入報告**：成功 N 筆、跳過 M 筆＋原因、缺哪些 PM 範本。
- **新增 `templates/README_範本放置說明.md`**：說明 11 份 Nokia 範本 + DK 範本要放哪、檔名要對應 `config.CHT_NOKIA_PM_FILES`。

### ⚠️ 你需要做的事
把固定範本 .xlsx 放進 `templates/cht_nokia/` 和 `templates/cht_dk/`（檔名見 README），
並 `git add templates/` 一起部署。否則 Nokia/DK 寫入仍會跳過所有人（但現在至少會明確告訴你原因）。

### 怎麼測
1. 放好範本後，到看板選好月份、填資料、按「寫入 → Nokia 工作天數表」。
2. 下載後應看到「✅ 寫入 N 筆」。若有人被跳過，報告會列出是「範本缺失 / PM 名稱不符 / 範本中找不到該員工」。

## 問題 2：核對記錄暫存區 ＋ 重複/遺漏提醒

### 改了什麼

**(A) 後端 forms 改成按月份分層保存（跨月偵測的前提）**
- 原本 `forms.json` 是 `{ emp_en: form }`，每人一份、新月份會蓋掉舊月份 → 沒有歷史可比。
- 改成 `{ period: { emp_en: form } }`，每月一份永久保留。
- **自動遷移**：載入舊格式時自動把資料搬到對應 period 底下，你現有的 P05 資料不會掉。
- 新增 `GET /api/forms/history/{emp_en}?before=2026-P05` → 取得該員工 P05 之前的所有月份記錄。
- 檔案：`services/sync_svc.py`、`routers/sync.py`。
- 前端 `components/Kanban.js` 存表單時帶上 `period`（`fullPeriod`），資料才會進對的月份桶。

**(B) 核對邏輯擴充（`src/verify.js`）**
- 每個核對項目（Travel/OT/NS/ESS）多回傳 `template_dates`（範本所有 approved 日期）、
  `keyed_dates`（你 key in 的日期）、`details`（key in vs 範本逐筆對照）。
- 新增**遺漏偵測**：範本有 approved、但你沒 key in 的日期。
- 新增**重複偵測**：本月 key in 的日期，若在過去月份（history）已申請過 → 標出是哪個月。
- `verifyEmployee(workbooks, empEn, empCn, tab, history)` 多吃一個 `history` 參數。

**(C) 核對記錄暫存區（`src/verifyStore.js` + `src/components/VerifyRecordsPage.js`）**
- 每次核對完，把「異常 / 遺漏 / 重複」攤平成一筆筆記錄，存到瀏覽器 localStorage（關閉、重整都還在）。
- 每筆記錄含：核對時間、月份、員工、項目、類型、**我 key in 的值 vs 範本記錄值**、備註。
- 頂部新增第三個分頁「**核對記錄**」（`App.js`），可依類型/月份/姓名篩選，單筆刪除、一鍵清空。
- 可**導出 JSON / CSV**（CSV 帶 BOM，Excel 開中文不亂碼）。
- 重新核對同員工同月份時，舊記錄自動覆蓋，不會堆疊。

### 怎麼測
1. 在看板填 P05 的 TA/ESS/OT/NS，同步到後端。
2. 按「核對」上傳對應 approval 檔，選員工核對。
3. 到「核對記錄」分頁，應看到異常/遺漏列出，每筆有「我 key in」對「範本記錄」。
4. 跨月重複：等你導入 P04 並 key in 後，下次核對 P05 時，若日期在 P04 出現過會標「重複」。
   （你目前只有 P05，所以重複偵測要等 P04 也 key in 後才有資料可比 — 這是預期行為。）

## 問題 3：檔案預覽（xlsx）沒反應

### 根本原因
`openFile` 點 xlsx 時設了 `modal = {type:"xlsx", url}`，但 `Kanban.js` 裡**沒有對應的 render 區塊**
（只有 pdf / eml），所以什麼都不會顯示。

### 改了什麼
- **前端 `api.js`**：修掉 `previewXlsxUrl` 的路徑編碼（斜線被編壞），新增 `previewXlsx()` 取得後端回的 HTML 表格。
- **前端 `components/Kanban.js`**：
  - `openFile` 的 xlsx 分支改成 `await previewXlsx(...)` 取 HTML 存進 `modal.html`。
  - 新增 `modal?.type === "xlsx"` 的 render 區塊，顯示後端解析好的表格。
- 後端 `/api/preview-xlsx/...`（原本就有）負責把 xlsx 轉成 HTML table。

### 怎麼測
在看板選員工 → 檔案分頁 → 點任一 xlsx 的預覽，應彈出表格內容（前 50 列）。
pdf 和 eml 預覽行為不變。

# 第二輪修正（依截圖回報）

## 問題 A：預覽「Employee not found」＋ 寫入「0 筆，名單共 0 人」

### 同一個根因
你的員工名單存在**該月份專屬名單**（`roster_2026-P05.json`），但有兩個地方讀的是**全域 `people.json`**（是空的）：
- 批量寫入：`batch_write_svc` 呼叫 `get_all()` 沒帶 period → 0 人。
- xlsx 預覽：`preview_xlsx` 呼叫 `find_by_name(emp_en)` 沒帶 period → 找不到員工。

已用「空全域 + 月份名單」重現此狀況，確認就是這個原因。

### 改了什麼
- **後端 `services/batch_write_svc.py`**：五個批量寫入函式全部改用 `get_all(period)`；
  原本沒有 period 參數的 wipro / project-f / nokia-cost 也加上 period 參數。
- **後端 `routers/batch.py`**：呼叫時把 `period` 傳進去；`preview_xlsx` 改成
  `find_by_name(emp_en, period) or find_by_name(emp_en)`（先查月份名單，再退回全域）。
- **前端 `api.js`**：`previewXlsx` / `previewFileUrl` 新增 `period` 參數。
- **前端 `components/Kanban.js`**：`openFile` 呼叫預覽時帶入 `fullPeriod`。

### 驗證
重現截圖情境（全域空、P05 名單有人）→ 修正後寫入 1/1（原本 0/0）、預覽回 200 顯示表格。

> 補充：若你希望全域名單和月份名單一致，可在「人員管理」確認名單有存到對的地方；
> 但即使只有月份名單，現在寫入和預覽都會正確讀取。

## 問題 B：核對加入日期範圍選擇

approval 檔常含整年資料，現在核對前可選「核對日期範圍」：
- 預設帶入當月（依 period，如 P05 → 該年 5/1～5/31）。
- 可自訂起訖，或按「整年/不限」清除範圍。
- 只有落在範圍內的 approval 紀錄會被核對，避免其他月份被誤判成「遺漏」。
- 核對記錄會多一欄「範圍」，導出 CSV 也含此欄。

### 改了什麼
- `src/verify.js`：新增 `dateInRange` / `rangeFilter`，`verifyEmployee` 及四個核對函式
  （Travel/OT/NS/ESS）都吃 `range` 參數並過濾 approval 紀錄。
- `src/components/Kanban.js`：核對 modal 上傳步驟加入日期範圍 UI，預設當月。
- `src/verifyStore.js` / `src/components/VerifyRecordsPage.js`：記錄與導出加入「範圍」欄。

### 驗證
用含整年（3月/5月/8月）approval 測試：不限範圍 → 8 天誤報遺漏；
限定 5 月 → 只核對 5/10–5/12，0 遺漏、狀態正常。

# 第三輪修正：遺漏偵測改為「上月+本月」

## 你的需求
原本「遺漏」是拿 approval 檔比對本月 key in，但 approval 檔常含整年資料，
導致其他月份（甚至未來月份）的核准日被誤報成遺漏。
正確邏輯應為：**Approval 檔（上月+本月範圍內） vs 我 key in（上月+本月）**。

## 改了什麼（皆前端）
- **核對範圍自動往前涵蓋上一個月**（`Kanban.js`）：
  你選的範圍（預設本月）會自動把起日往前拉到「上個月 1 號」，approval 檔只看這兩個月。
  例：核對 P05 預設範圍 5/1–5/31 → 實際過濾範圍變成 4/1–5/31。
- **遺漏排除上月已 key in 的日期**（`verify.js`）：
  `findMissing` 現在同時扣掉「本月 tab」和「上月（歷史記錄）」key 過的日期，
  新增 `priorKeyedDates()` 從後端歷史抓上月 key in 的日期。
- UI 文字、核對記錄備註、範圍欄都更新為「含上月」。

## 效果（已實測你描述的情境）
上月 key in 04/01–04/24、本月 key in 05/01–05/11，approval 另有核准 04/25–04/28（從未 key in）、
以及未來的 06/10–06/12：
- ✅ 只有 **04/25–04/28** 被標遺漏（真正跨月漏掉的）
- ✅ 04/01–04/24（上月已 key in）、05/01–05/11（本月已 key in）都不誤報
- ✅ 06/10–06/12（超出範圍的未來月份）完全不顯示 → 解決截圖的誤報問題
